# 高级功能说明

## 1. MODBUS 命令外部化 (`modbus_config.json`)

### 功能

传感器读取命令从代码中抽离到 JSON 配置文件，修改命令**无需重新编译**程序。

### 什么时候用

- 更换了不同型号的传感器，寄存器地址不同
- 传感器固件升级导致命令格式变化
- 需要新增自定义读取命令
- 把程序发给别人用，他的传感器配置不同

### 文件位置

`bin\Debug\modbus_config.json`（编译后自动复制，`PreserveNewest` 模式不覆盖已修改的文件）

### 格式

```json
{
  "commands": [
    { "index": 0, "name": "一次读温湿(浮点)", "bytes": "01,03,00,00,00,04,44,09" },
    { "index": 1, "name": "一次读温湿(整型)", "bytes": "01,03,00,80,00,04,45,E1" }
  ]
}
```

| 字段 | 说明 |
|------|------|
| `index` | 对应下拉框"读取模式"的序号（0-9） |
| `name` | 备注名称，不影响功能 |
| `bytes` | MODBUS RTU 命令的十六进制字节，逗号分隔，CRC 需预先计算 |

### 使用方法

1. 打开 `bin\Debug\modbus_config.json`
2. 修改对应 `index` 的 `bytes` 值
3. 保存文件
4. **重启程序**即生效

### 容错

如果 JSON 文件不存在或格式错误，程序自动回退到内置默认命令，不影响正常运行。

### 实际例子

```
场景：传感器地址从 01 改为 02

修改前：01,03,00,00,00,04,44,09
修改后：02,03,00,00,00,04,77,D9     ← 地址改 02，CRC 重新计算
```

## 2. 单元测试 (`Tests\TestRunner`)

### 功能

自动验证核心算法（CRC 校验、命令生成、数据解析）是否正确。

### 什么时候用

- 修改了 `modbus_config.json` 中的命令后，验证 CRC 是否正确
- 修改了 `getSensorData` 数据解析逻辑后，验证解析结果
- 代码重构后，确认没有破坏现有功能
- 发布新版本前的回归测试

### 运行方法

```bash
# 1. 进入项目根目录
cd TempHumidityMonitor

# 2. 编译测试
MSBuild Tests\TempHumidityMonitor.Tests.csproj /p:Configuration=Debug

# 3. 运行
bin\Tests\TempHumidityMonitor.Tests.exe
```

### 预期输出

```
=== MODBUS CRC 测试 ===
  [PASS] CRC-16: 01 03 00 00 00 04 → 44 09
  [PASS] CRC-16 标准测试向量
  [PASS] 有效帧 CRC 验证通过
  [PASS] 损坏帧 CRC 验证拒绝
  [PASS] 无效帧头被拒绝
  [PASS] 错误长度被拒绝

=== 命令生成测试 ===
  [PASS] 命令0长度=8
  [PASS] 命令0 CRC正确
  ...（共 20 条）

=== 数据解析测试 ===
  [PASS] 浮点温度 25.0 (实际: 25)
  [PASS] 整型温度 25.0 (实际: 25)

=== 结果: 28 通过, 0 失败 ===
```

### 测试内容

| 测试组 | 验证内容 | 数量 |
|--------|----------|------|
| MODBUS CRC | CRC-16 计算正确性、有效帧/损坏帧/无效帧头/错误长度 | 6 |
| 命令生成 | 10 条命令长度和 CRC 校验 | 20 |
| 数据解析 | 浮点温度和整型温度解析 | 2 |

### 实际例子

```
场景：修改了 getSensorData 中的字节序

改之前测试：浮点温度 25.0 → [PASS]
改之后测试：浮点温度 25.0 实际: 7.18E-41 → [FAIL]
                                    ↑ 立刻知道字节序搞反了
```
